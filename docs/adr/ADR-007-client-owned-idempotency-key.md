# ADR-007: Idempotency Key Owned by Client

**Status:** Accepted
**Date:** 2026-02-26

## Context

To prevent double-charging on network retries, purchase requests must be idempotent. The idempotency key must be tied to a specific purchase intention, not generated by the server.

## Decision

The client generates the idempotency key (a UUID) before sending the request and includes it in the `Idempotency-Key` header.

## Reasoning

Server-generated keys cannot solve the retry problem — if the server generates the key, the client doesn't know it on a retry. The client must own the key so it can reuse it across retries of the same purchase intention.

The contract:

- One UUID per purchase intention, generated client-side before the first request
- Same key on every retry of that same purchase
- New UUID for every new purchase intention

The database enforces uniqueness via a `UNIQUE` constraint on `purchases.idempotency_key`. Even if two concurrent retries bypass the application-level check, only one will succeed at the INSERT — the other fails with a constraint violation, caught and returned as 409.

## Consequences

- Client is responsible for key generation and storage
- Server is stateless with respect to idempotency — it just checks the database
- Works correctly even under concurrent retries without distributed locks
